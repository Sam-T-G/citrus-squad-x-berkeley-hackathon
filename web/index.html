What we are building

A teammate wears a belt with four tap motors (far-left, left, right, far-right) and a phone mounted on the chest facing forward. The phone reads the route and the wearer's heading, works out which way they need to turn, and taps the matching side of the belt. Its LiDAR and camera watch for obstacles and fire a stronger tap when something is close. No audio, no looking at a screen.

How the pieces connect

📱 Phone brain

  • GPS + compass (heading)
  • LiDAR depth (obstacles)
  • Camera + vision (objects)
  • Maps + route logic
  • Decides one cue, 10x/sec
→
Wi-Fi (UDP)
4-byte LC2 packet
wireless, no cable

🔌 ESP32 muscle

  • Listens for LC2 packets
  • Turns events into tap patterns
  • Goes quiet if the link drops
→
PWM signal
(wires)

🪢 4 servos taps

  • Front, left, right, back
  • Left/right rotate, front forward, back proximity
  • Tap on the wearer's torso
The only cables are power into the ESP32 and the four servo wires. The phone talks to the belt over the air, so nothing plugs into the phone during use.

How a cue gets decided

Every tenth of a second the phone runs one loop:

route or simulator  ──┐
                      ├─►  decide one cue  ──►  belt (taps)
LiDAR + camera  ──────┘     safety beats        audio (optional)
   (hazards)               direction            demo screen

Safety wins: if something is close, the belt warns about the obstacle instead of the turn, then goes back to directions once the path is clear. Whatever the phone decides goes out to every output at once, so the belt, the on-screen demo, and the (optional) audio all stay in sync.

The phone app

Operate

The clean demo screen. One big cue (turn left, slight right, obstacle, arrived) with connect, calibrate, and run. The phone buzzes in time with the belt, so it reads even without the hardware.

Demo

The visualizer for the audience: a map with position and route, the camera with vision boxes drawn on it, and the live LiDAR depth bands. Shows what the phone sees.

Diagnostics

The bench console: every sensor read out raw, the belt link, the nav math, a route simulator, the Maps usage counters, and a thermal soak test.

Run it on your own phone

Each of us runs our own instance. Full guide is in RUNNING.md. The short version:

git clone <repo> && cd citrus-squad-x-berkeley-hackathon
./ios/setup.sh                 # installs XcodeGen, makes your local config, builds the project
# edit ios/Local.xcconfig: set DEVELOPMENT_TEAM and APP_BUNDLE_ID (com.yourname.CitrusSquad)
./ios/setup.sh && open ios/CitrusSquad.xcodeproj
# pick your iPhone, press Cmd-R

You do not need the belt or a Maps key to develop. Open Diagnostics → Navigation, tap Load demo route, switch to Simulate, and hit Run sim. The whole route-to-cue path runs in software.

Plug points for the vision work

The app is built so Cole and Josh's model drops in without touching the core loop:

// 1. Draw boxes on the Demo camera (normalized 0..1 rectangles)
model.detections.update([
  Detection(label: "person", confidence: 0.92, box: CGRect(x: 0.3, y: 0.3, width: 0.3, height: 0.5))
])

// 2. Fire a belt warning when a person is in the path
model.vision.report(kind: .person, side: .centerMass, distanceMeters: 1.8)

// 3. Need raw frames? Attach an output to the shared session
model.camera.session   // add an AVCaptureVideoDataOutput here
One hardware note for the vision work: the rear camera is exclusive, so ARKit LiDAR depth and the camera cannot both run at the same instant yet. Moving depth to AVFoundation would let one session feed both. It is on the list.

Decisions and blockers

1. How does the vision reach the app? Cole's CV runs off-device today: the phone streams RGB + LiDAR frames over WebSocket to a laptop running the Python server, which streams detections back. The app's current seams (DetectionStore, VisionHazardSource) are on-device. Pick one path to close the gap:
  • Server path: add a WebSocket sender and receiver to the Swift app. The frame format and the /haptics JSON are already defined in Cole's HANDOFF. Fast to wire, but the demo then depends on a laptop and the venue Wi-Fi.
  • On-device path: run Cole's YOLOv8n as CoreML on the phone (his export script is ready) and feed DetectionStore directly. No laptop, a bit more setup.
  • Roles: Cole's notes have Josh owning the iOS/Swift app, but that is Sam's lane. Confirm: Sam = app, Cole = CV, Josh = which CV slice, Angelo = belt.
  • Firmware: Angelo to adopt the front/back/left/right motor proposal (proposals/) and push a branch.
  • Wire format: confirm Cole's /frames layout and the real LiDAR depth resolution (256x192 assumed) against the live iPhone stream.
